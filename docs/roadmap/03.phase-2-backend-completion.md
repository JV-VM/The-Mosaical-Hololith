# Phase 2: Backend Completion Backlog

## Goal

Finish the backend so it can support the full frontend modulith without relying on MVP shortcuts, manual operator steps, or placeholder integrations.

This phase is still backend-only. It should end with a stable API and operational baseline that Phase 3 can consume without major contract churn.

## Current baseline

The backend already provides working modules for:

- auth
- tenants
- stores
- pages
- catalog
- tags
- discovery
- plans
- analytics

The main Phase 2 gaps are explicit in the codebase today:

- media is only a stub in [media.module.ts](/home/vihkctormartim/The-Mosaical-Hololtih/mosaical-hololith-backend/src/media/media.module.ts)
- billing upgrades are still direct plan mutation without payment workflow in [plans.controller.ts](/home/vihkctormartim/The-Mosaical-Hololtih/mosaical-hololith-backend/src/plans/plans.controller.ts)
- auth currently exposes register, login, refresh, and me only in [auth.controller.ts](/home/vihkctormartim/The-Mosaical-Hololtih/mosaical-hololith-backend/src/auth/auth.controller.ts)
- admin-like tag creation is still an MVP path mixed into the tag controller in [tags.controller.ts](/home/vihkctormartim/The-Mosaical-Hololtih/mosaical-hololith-backend/src/tags/tags.controller.ts)

## Phase 2 exit criteria

Phase 2 is complete only when all of the following are true:

- media upload, metadata persistence, and media attachment flows are implemented
- billing no longer relies on direct upgrade endpoints without provider-backed state
- account lifecycle flows exist for verification, password reset, and invitation acceptance
- admin-only backend capabilities are separated from MVP shortcuts and have explicit authorization
- API validation, pagination, and error contracts are consistent enough for frontend consumption
- backend tests cover new critical flows at unit and e2e level
- root verification remains green after the work

## Delivery rules

- Avoid broad schema churn unless it is required by a specific ticket.
- Keep new external providers behind adapters so Render deployment and future provider swaps stay manageable.
- Prefer additive migrations and backward-compatible API changes until the frontend modulith is consuming the new endpoints.
- Split provider integration from product workflow where possible. For example, build billing domain state before wiring payment webhooks.

## Workstreams

### W1. Contract and domain hardening

This workstream reduces API ambiguity before more feature-specific code lands.

#### P2-01: Standardize API error envelopes

Scope:

- define one backend error response shape
- apply it consistently across auth, stores, pages, catalog, tags, plans, and analytics
- document which errors are user-facing validation failures vs authorization vs missing resources

Dependencies:

- none

Acceptance criteria:

- error responses are predictable enough for a shared frontend API client
- 4xx and 5xx responses no longer vary by module without reason

#### P2-02: Standardize pagination and list query contracts

Scope:

- choose one pagination style for list endpoints
- align discovery, tags, catalog, stores, and admin-facing list endpoints
- define common query parameters for page size, cursor or offset, sort, and filters

Dependencies:

- P2-01

Acceptance criteria:

- list endpoints use the same response structure
- frontend list views can reuse one pagination adapter

#### P2-03: Tighten authorization boundaries

Scope:

- audit producer vs tenant admin vs platform admin behaviors
- remove controller-level ad hoc role checks where a reusable guard or policy is more appropriate
- make admin-only operations explicit instead of “MVP-only” paths

Dependencies:

- none

Acceptance criteria:

- role-dependent operations are guarded consistently
- phase 2 admin endpoints do not rely on environment checks as the primary control

### W2. Media subsystem

This turns the empty media module into a real backend capability.

#### P2-10: Design media domain model and storage contract

Scope:

- define media asset records, ownership, and attachment rules
- decide whether product media and page media share one asset model
- define supported file classes, size limits, and MIME validation
- define S3-compatible adapter boundary for object storage

Dependencies:

- P2-01
- P2-03

Acceptance criteria:

- Prisma schema additions are documented before implementation
- storage interactions are isolated behind a provider adapter

#### P2-11: Implement media upload session flow

Scope:

- create endpoints for upload intent or presigned upload flow
- persist pending or uploaded media metadata
- validate tenant ownership and target entity ownership

Dependencies:

- P2-10

Acceptance criteria:

- producers can request upload capability without direct storage credentials
- backend can track uploaded asset metadata independently of the frontend

#### P2-12: Implement media attach, detach, and lookup flows

Scope:

- attach media assets to products and later page content if needed
- support removal and safe replacement
- expose asset metadata in product and store-facing payloads

Dependencies:

- P2-11

Acceptance criteria:

- catalog payloads no longer depend on arbitrary JSON blobs alone for media lifecycle
- orphaned assets are detectable

#### P2-13: Add media cleanup and failure handling

Scope:

- define cleanup path for abandoned uploads and deleted references
- add retry-safe deletion semantics
- prepare worker or cron follow-up if cleanup should be async later

Dependencies:

- P2-11
- P2-12

Acceptance criteria:

- asset lifecycle has a documented cleanup strategy
- delete operations do not create silent storage leaks

### W3. Billing and subscription lifecycle

This replaces the current plan mutation shortcut with provider-backed billing state.

#### P2-20: Separate plan catalog from subscription state transitions

Scope:

- formalize internal billing states and allowed transitions
- separate “available plans” from “active subscription”
- define upgrade, downgrade, cancel, and reactivation semantics

Dependencies:

- P2-01
- P2-03

Acceptance criteria:

- plan and subscription behavior is represented in service logic, not only controller responses
- direct unrestricted plan switching is removed from the product path

#### P2-21: Add payment provider adapter and configuration model

Scope:

- introduce billing provider abstraction
- define customer, subscription, checkout session, and webhook event mappings
- wire configuration through environment variables appropriate for Render

Dependencies:

- P2-20

Acceptance criteria:

- provider-specific code is isolated
- local development can run with a stub or test provider mode

#### P2-22: Implement checkout and subscription management endpoints

Scope:

- create endpoint flow for starting checkout or plan change
- expose current billing status and actionable next steps
- prevent direct mutation of subscription state from UI-facing endpoints

Dependencies:

- P2-21

Acceptance criteria:

- frontend can initiate billing flow without privileged direct plan mutation
- subscription status reflects provider-backed workflow state

#### P2-23: Implement billing webhooks and reconciliation

Scope:

- receive and verify provider webhook events
- update subscription state idempotently
- add reconciliation path for missed or delayed events

Dependencies:

- P2-21
- P2-22

Acceptance criteria:

- duplicate webhook deliveries are safe
- billing state no longer depends on one synchronous request completing successfully

### W4. Account lifecycle and email-triggered flows

This turns auth into a real account system instead of a token-only login surface.

#### P2-30: Introduce email provider adapter and notification primitives

Scope:

- define email adapter boundary
- add provider configuration model
- add templated notification primitives for verification, reset, and invite flows

Dependencies:

- P2-01

Acceptance criteria:

- auth and invite flows can trigger emails without hard-coding provider logic into controllers

#### P2-31: Add email verification flow

Scope:

- persist verification token state
- expose request and confirm endpoints
- decide whether unverified users can create tenants, publish stores, or invite others

Dependencies:

- P2-30

Acceptance criteria:

- user verification state exists and can be enforced where needed

#### P2-32: Add password reset flow

Scope:

- create request-reset and confirm-reset endpoints
- expire or invalidate reset tokens correctly
- invalidate prior refresh-token trust after password reset

Dependencies:

- P2-30

Acceptance criteria:

- users can recover accounts without manual operator intervention

#### P2-33: Add tenant invitation flow

Scope:

- allow tenant admins to invite users by email
- support invite acceptance for existing and new users
- map invite role to membership creation safely

Dependencies:

- P2-03
- P2-30

Acceptance criteria:

- tenant team expansion no longer requires direct database or manual backoffice steps

### W5. Platform admin backend surface

This separates internal governance capability from producer-facing backend logic.

#### P2-40: Define platform admin role model

Scope:

- decide whether platform admin is a separate identity class, tenant-independent role, or both
- define how admin auth and impersonation-safe access work

Dependencies:

- P2-03

Acceptance criteria:

- platform admin authorization is explicit and not overloaded onto tenant membership alone

#### P2-41: Move tag governance out of MVP seed paths

Scope:

- replace current admin tag creation shortcut with explicit platform admin endpoints
- preserve public tag listing and tenant-scoped assignment behavior
- define audit-friendly mutation responses for taxonomy changes

Dependencies:

- P2-40

Acceptance criteria:

- tag governance no longer depends on `NODE_ENV` checks or controller-local admin assertions

#### P2-42: Add moderation and inspection endpoints

Scope:

- add backend support for store inspection, product inspection, and content moderation decisions
- include enough metadata for a future admin UI
- define any needed status flags or moderation notes

Dependencies:

- P2-40

Acceptance criteria:

- platform operators can investigate tenant content through API endpoints instead of raw database access

#### P2-43: Add audit logging for sensitive admin and billing actions

Scope:

- log critical administrative and billing state changes
- capture actor, target, action, and timestamp
- ensure logs are structured enough for later UI/reporting use

Dependencies:

- P2-22
- P2-40

Acceptance criteria:

- sensitive changes are traceable without depending on infrastructure logs alone

### W6. Test coverage and release confidence

This workstream converts the new backend scope into something Phase 3 can trust.

#### P2-50: Add unit coverage for each new domain service

Scope:

- cover media service behavior
- cover billing state transitions and provider adapter behavior
- cover auth lifecycle token and email flows
- cover admin authorization and audit behavior

Dependencies:

- all feature workstreams as implemented

Acceptance criteria:

- service-level domain rules are test-backed

#### P2-51: Add end-to-end flows for critical journeys

Scope:

- registration plus verification
- password reset
- invite acceptance
- plan checkout initiation and webhook-driven subscription update
- media upload intent and media attachment

Dependencies:

- P2-11
- P2-22
- P2-23
- P2-31
- P2-32
- P2-33

Acceptance criteria:

- critical product journeys are verifiable from API entrypoint to persisted state

#### P2-52: Publish Phase 2 backend API change log

Scope:

- document new endpoints, changed contracts, and migration notes for the upcoming frontend phase
- identify deprecated MVP-only routes and their replacements

Dependencies:

- all finalized feature tickets

Acceptance criteria:

- Phase 3 can build against a stable and documented backend contract set

- frontend UI implementation
- full docs site publishing
- Render worker and cron production rollout
- advanced observability stack beyond what is needed to support backend implementation
- custom domain verification flows

Those belong to later phases even if some provider adapters are introduced now.
