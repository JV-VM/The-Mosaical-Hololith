# Phase 1: Target Architecture and Render Topology

## Decision summary

The target platform is a hybrid decoupled monorepo deployed on Render.

- The frontend is a single modulith application.
- The backend is a decoupled NestJS API service.
- Both live in the same repository but deploy independently.
- Docker is the deploy artifact for application services.
- Render is the primary deployment platform.

## Architecture principles

- Keep one frontend codebase, not three separate frontend repos.
- Keep frontend and backend decoupled at runtime through typed HTTP contracts.
- Keep backend domains modular and independently testable.
- Keep infrastructure defined in repo, not only in the Render dashboard.
- Keep documentation versioned with the code.

## Backend location

The backend lives at `apps/api/`.

- root scripts target `apps/api/`
- Render builds `tmh-api` from `apps/api/`
- local Docker Compose builds the API from `apps/api/Dockerfile`

## Frontend modulith design

The frontend is one deployable application with internal route and module boundaries.

### Route surfaces

- Public hub: `/`
- Discovery: `/explore`, `/tags/:slug`, `/stores/:slug`, `/stores/:slug/p/:productSlug`
- Producer dashboard: `/dashboard/*`
- Platform admin: `/admin/*`
- Auth and account: `/auth/*`, `/account/*`
- Docs, if embedded: `/docs/*`

### Internal frontend modules

- `public-hub`
- `catalog`
- `stores`
- `pages`
- `auth`
- `dashboard`
- `analytics`
- `billing`
- `admin`
- `shared-ui`
- `shared-data`

### Frontend boundary rules

- Public pages must not import admin-only state or UI.
- Admin modules must sit behind explicit role gates.
- Shared UI components stay presentation-first and domain-agnostic.
- All backend access goes through a typed API client layer.
- Feature flags and environment reads must be centralized.

## Backend service design

The backend remains a separate service responsible for:

- authentication
- tenancy
- stores
- pages
- catalog
- tags
- discovery
- plans and quotas
- analytics
- media API
- integration webhooks

Phase 1 does not replace the backend architecture. It formalizes how the frontend consumes it.

## Runtime contract

- Frontend communicates with backend over HTTPS JSON APIs only.
- Auth state is managed by the frontend but enforced by the backend.
- Shared types must be generated or carefully mirrored, not tightly runtime-coupled.
- API versioning should be explicit before frontend feature expansion accelerates.

## Render deployment topology

### Environment set

- `development`
- `test`
- `staging`
- `production`

### Services

### 1. `web`

Type: Render web service  
Artifact: Docker image  
Responsibility: frontend modulith for public, dashboard, and admin surfaces

Suggested domains:

- `www.example.com`
- `app.example.com`
- `admin.example.com`

These can all point to the same `web` service if the frontend handles hostnames and route segmentation cleanly.

### 2. `api`

Type: Render web service  
Artifact: Docker image  
Responsibility: NestJS backend API

Suggested domain:

- `api.example.com`

### 3. `worker`

Type: Render background worker  
Artifact: Docker image  
Responsibility:

- media processing
- email dispatch
- webhook retries
- analytics aggregation
- other async jobs

Add this when Phase 7 starts if synchronous request handling becomes too heavy.

### 4. `cron`

Type: Render cron job  
Artifact: Docker image  
Responsibility:

- nightly cleanup
- analytics compaction
- scheduled sync jobs
- subscription reconciliation

### 5. `postgres`

Type: Render managed PostgreSQL  
Responsibility:

- primary application data
- Prisma migrations

### 6. `redis` (optional but likely)

Type: Render Key Value  
Responsibility:

- queueing
- caching
- rate-limit state if externalized later

### 7. `docs`

Type: Render static site or web service  
Responsibility:

- published technical docs
- user docs
- runbooks and deployment guides

If the team wants documentation versioned and discoverable separately, keep this as a dedicated service.

## External services still required

Render is the deployment platform, but some production dependencies will still be external:

- S3-compatible object storage for media
- payment provider
- transactional email provider
- DNS provider for custom-domain verification
- optional error monitoring SaaS

## Render repo-owned infrastructure

Phase 1 should define the following repo-owned files:

- `render.yaml`
- `infra/render/env-matrix.md`
- `apps/api/README.md`
- `apps/web/README.md`
- root deployment guide pointing to the Render flow

## Environment variable classes

Define variables in groups from the beginning:

- shared non-secret config
- frontend public config
- backend runtime secrets
- migration-only secrets
- integration secrets
- test-only config

Do not let frontend-only public vars and backend private vars share the same naming style or storage path.

## Acceptance criteria

Phase 1 is complete only when all items below are true:

- The monorepo target structure is fixed.
- The backend relocation strategy is decided.
- The frontend modulith boundaries are documented.
- The Render service topology is documented.
- Domain and environment strategy are documented.
- External dependencies are explicitly listed.
- Phase 2 implementation can start without architecture ambiguity.
