# Phase 0: Repo Stabilization

## Purpose

Stabilize the current repository before any large-scale frontend or infrastructure work begins.

This phase exists because the current tree has real code but also clear environment drift:

- the repo is backend-first instead of monorepo-ready
- package manager execution is inconsistent
- local ownership and permission issues exist
- the test database path is not reliably bootstrapped
- the deployment story is fragmented across docs and service files

## Success criteria

Phase 0 is complete only when all items below are true:

- A new engineer can clone the repo and get to a green local build using one documented bootstrap path.
- `install`, `lint`, `build`, `unit tests`, and `e2e tests` run from a clean checkout.
- Docker local development is documented and reproducible.
- The repository is ready to accept `apps/`, `packages/`, and `infra/` additions without structural churn.
- CI enforces the same commands used locally.

## Workstreams

### 0.1 Toolchain standardization

- Pick one package manager for the entire repo.
- Enforce the version with Corepack and a documented Node version.
- Add a root workspace file once Phase 1 starts.
- Remove mixed `npm` vs `pnpm` instructions from docs.
- Add a root bootstrap command and a root verification command.

### 0.2 Repository ownership and permissions

- Fix Git safe-directory friction for local contributors.
- Fix write permissions for generated artifacts such as `dist/`.
- Ensure Docker and local processes do not leave root-owned files in the workspace.
- Define an ownership policy for generated files and containers.

### 0.3 Environment and secrets baseline

- Add `.env.example` at the backend root.
- Add `.env.test.example`.
- Document required variables, defaults, and local-only values.
- Split runtime env, migration env, test env, and future frontend public env clearly.
- Define environment naming for `development`, `test`, `staging`, and `production`.

### 0.4 Test and build reliability

- Make `build` run cleanly from source without manual cleanup.
- Make unit tests green from a clean checkout.
- Make e2e tests bring up their own database path through Docker.
- Add explicit migration and seed commands for test bootstrap.
- Add a smoke test command for local API verification.

### 0.5 CI baseline

- Add or repair CI to run install, lint, build, unit tests, and e2e tests.
- Fail CI on drift between lockfile, Prisma schema, and generated client.
- Cache dependencies correctly.
- Publish test and coverage artifacts if useful.

### 0.6 Repo preparation for monorepo transition

- Decide whether `apps/api/` is renamed to `apps/api/` in Phase 1 or migrated in place first.
- Add a root-level conventions document for future app and package placement.
- Reserve directories for `apps/`, `packages/`, and `infra/`.
- Ensure existing relative paths will not block the move.

### 0.7 Documentation cleanup

- Update the root README to describe the current state honestly.
- Separate current-state docs from target-state docs.
- Add one canonical development path instead of multiple competing instructions.

## Concrete backlog

### Must do first

- Choose and lock the package manager.
- Fix `dist/` permission problems.
- Fix local e2e bootstrap and DB connectivity.
- Fix install/build/test instructions.

### Must do before Phase 1 closes

- Add env templates.
- Add or repair CI.
- Decide the backend relocation strategy.
- Add root workspace preparation.

### Can follow immediately after Phase 1 starts

- Move service scripts to root orchestration commands.
- Add frontend placeholders once the workspace exists.

## Expected artifacts

- Root `README.md` update
- Root workspace config
- Backend env examples
- Stable local Docker test flow
- CI workflow that matches the documented local flow
- Decision note on `apps/api/` migration

## Risks if skipped

- Frontend work starts against an unstable repo and slows everything down.
- Render deployment work happens before build/test behavior is deterministic.
- Ownership and permission drift continues to break CI and local builds.
- Team members create parallel bootstrap instructions that diverge.

## Exit checklist

- Local install works.
- Local build works.
- Unit tests pass.
- E2E tests pass.
- Migration flow is documented and reproducible.
- CI mirrors local commands.
- The repo is structurally ready for Phase 1.
