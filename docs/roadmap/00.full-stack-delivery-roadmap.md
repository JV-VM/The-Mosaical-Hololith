# Full-Stack Delivery Roadmap

## Objective

Finish The Mosaical Hololith as a production-grade full-stack platform deployed on Render, with:

- a decoupled NestJS backend API
- a single frontend modulith application
- Docker-based deploys for application services
- operational documentation, developer documentation, and user-facing documentation

## Locked decisions

- Deployment platform: Render
- Repository style: hybrid decoupled monorepo
- Frontend architecture: modulith
- Backend architecture: modular monolith API
- Runtime boundary: frontend and backend deploy independently and communicate over HTTP/JSON
- Delivery artifact: Docker images for deployable app services

## Phase sequence

### Phase 0: Repo Stabilization

Goal: make the current repository reproducible, testable, and safe to evolve.

Core subtasks:

- Normalize repository ownership, permissions, and local bootstrap.
- Standardize the package manager and workspace strategy.
- Repair local build, lint, unit-test, and e2e-test flows.
- Add environment templates and local Docker development flows.
- Make CI enforce the same path engineers use locally.

Primary output:

- A clean engineering baseline for all later work.

### Phase 1: Target Architecture and Monorepo Layout

Goal: define the final repo structure, app boundaries, and Render topology before feature expansion.

Core subtasks:

- Define the monorepo layout for `web`, `api`, shared packages, infra, and docs.
- Define frontend modulith route and module boundaries.
- Define backend API ownership and integration contracts.
- Define Render service topology, domains, environments, and deployment flow.
- Define external dependencies that Render does not natively provide.

Primary output:

- One target architecture document that future implementation follows.

### Phase 2: Backend Completion

Goal: finish backend capabilities that are currently partial, stubbed, or MVP-only.

Core subtasks:

- Finish media management and upload lifecycle.
- Replace MVP billing stubs with a real subscription workflow.
- Add email verification, password reset, invites, and notification triggers.
- Add admin-safe moderation and reporting endpoints.
- Harden authorization, validation, error contracts, and pagination.
- Close test gaps and raise e2e coverage.

Primary output:

- A backend API ready to support the full product surface.

### Phase 3: Frontend Foundation

Goal: create the single frontend app and its shared internal foundations.

Core subtasks:

- Scaffold the frontend app and workspace configuration.
- Create shared design tokens, layout primitives, and auth/session handling.
- Build a typed API client and validation adapters.
- Establish route groups for public, dashboard, and admin areas.
- Add observability hooks, error boundaries, and loading conventions.

Primary output:

- A stable base for all UI surfaces.

### Phase 4: Public Hub

Goal: ship the public-facing discovery and storefront experience.

Core subtasks:

- Build landing pages, explore, tags, store pages, product pages, and public CMS pages.
- Add SEO metadata, sitemap generation, structured data, and canonical links.
- Add analytics tracking and public performance budgets.
- Add responsive, accessible UI states and search/filter persistence.

Primary output:

- A usable public web experience for visitors.

### Phase 5: Producer Dashboard

Goal: give producers and tenant admins a complete store management surface.

Core subtasks:

- Implement onboarding, tenant selection, and account settings.
- Implement store, page, product, tag, plan, and analytics management.
- Add validation, draft/publish workflows, and safe destructive actions.
- Add usage and quota visibility tied to plan enforcement.

Primary output:

- A complete producer-facing control surface.

### Phase 6: Platform Admin

Goal: provide internal governance and moderation tooling.

Core subtasks:

- Implement admin login, route guards, and audit visibility.
- Build taxonomy, policy, moderation, store inspection, and reporting screens.
- Add search, filters, impersonation-safe tooling, and operational safeguards.

Primary output:

- A platform operations surface that removes direct DB dependence.

### Phase 7: External Integrations

Goal: connect the platform to production dependencies.

Core subtasks:

- Add S3-compatible object storage for media.
- Add payment provider integration and webhook processing.
- Add transactional email provider integration.
- Add domain verification and DNS-related flows.
- Add Redis-backed queueing and caching where needed.

Primary output:

- Product-critical integrations are real, not manual or stubbed.

### Phase 8: Render Delivery and Environments

Goal: make staging and production deployable, repeatable, and observable on Render.

Core subtasks:

- Add Dockerfiles for deployable services.
- Add `render.yaml` and environment groups.
- Configure Render web services, workers, cron jobs, Postgres, and optional Redis.
- Configure custom domains, TLS, previews, and migration flow.
- Add post-deploy smoke tests and rollback procedures.

Primary output:

- A repeatable Render deployment model.

### Phase 9: Security, Reliability, and Observability

Goal: reduce production risk before launch.

Core subtasks:

- Add metrics, logging, tracing, error monitoring, and uptime checks.
- Add CSP, secret management, dependency scanning, and image scanning.
- Validate DB backups, restore procedures, and migration rollback plans.
- Perform load testing, rate-limit tuning, and connection-pool tuning.

Primary output:

- Production reliability controls and incident visibility.

### Phase 10: Documentation

Goal: ship extensive documentation as a maintained product asset.

Core subtasks:

- Write local setup, architecture, API, deployment, runbook, and release docs.
- Publish product user guides for visitors, producers, tenant admins, and platform admins.
- Publish operational runbooks for incidents and recovery.
- Publish diagrams, ADRs, and environment matrices.

Primary output:

- Docs that let a new engineer or operator succeed without tribal knowledge.

### Phase 11: Launch Readiness and Production Rollout

Goal: move from implementation-complete to safely operating in production.

Core subtasks:

- Seed staging with realistic data.
- Run UAT across all core journeys.
- Validate monitoring, backups, webhooks, and rollback.
- Perform a staged cutover and monitor the first production window.

Primary output:

- A live, supportable product.
