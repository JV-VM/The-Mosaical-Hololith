# System Architecture

## Intent

The Mosaical Hololith is being built as a hybrid decoupled platform:

- one frontend modulith application
- one backend API service
- shared packages inside the same repository
- independent runtime deployment boundaries

This is not a distributed microservice system. It is a monorepo with deliberate runtime separation.

## Architectural shape

### Frontend

The frontend is a single deployable web application that contains three route surfaces:

- public hub
- producer dashboard
- platform admin

The frontend is a modulith because:

- shared UI, auth, routing, and API client concerns should live in one codebase
- public, dashboard, and admin experiences need consistent behavior and design primitives
- independent frontend repos would add operational complexity before the product warrants it

The frontend remains internally modular through route groups and feature boundaries.

### Backend

The backend remains a modular monolith API implemented with NestJS.

It owns:

- auth
- tenancy
- stores
- pages
- catalog
- tags
- discovery
- plans and quotas
- analytics
- media APIs
- future webhooks and integration orchestration

### Runtime boundary

The frontend and backend are decoupled at runtime:

- frontend communicates with backend over HTTPS and JSON
- frontend never imports backend runtime code
- backend never depends on frontend deployment state
- cross-boundary contracts are expressed through typed clients, DTOs, and OpenAPI-style documentation

## Frontend modulith boundaries

The future `apps/web` application should be organized around internal feature modules instead of multiple deployed apps.

Minimum feature areas:

- `public-hub`
- `discovery`
- `stores`
- `catalog`
- `pages`
- `auth`
- `dashboard`
- `analytics`
- `billing`
- `admin`
- `shared-ui`
- `shared-data`

Boundary rules:

- public routes must not depend on admin-only state
- admin routes must be protected by explicit role gates
- all network access must go through a typed API client layer
- environment-variable reads must be centralized
- shared UI packages must remain domain-agnostic where possible

## Render deployment model

Phase 1 uses Render as the deployment platform of record.

Near-term deployable services:

- `api` web service
- `postgres` managed database

Planned services after the frontend exists:

- `web` web service
- `worker` background worker
- `cron` scheduled job
- optional `keyvalue` instance
- optional docs site

## External dependencies outside Render

Render is the hosting platform, not the full product stack.

Planned external dependencies:

- S3-compatible object storage
- payment provider
- transactional email provider
- DNS provider for custom domains
- optional error-monitoring SaaS

## Phase 1 implementation rule

Architecture artifacts should land before repository relocation or frontend implementation accelerates.

That means:

- define the target layout
- define the Render topology
- define the backend relocation strategy
- reserve the app/package directories
- keep the current backend green while the target structure is introduced
